Retail Anomaly Explainer

An AI assistant for translating data signals

Introduction

Built as part of a graduate AI/ML lab exercise, this project applies Design Thinking to a real problem I face at work: translating anomaly detection output into language a non-technical stakeholder can act on.

Description

A conversational AI agent that takes a plain-language description or a short data snippet of a business anomaly, asks clarifying questions when the input is vague, and returns a ranked, jargon-free explanation of likely causes along with what to verify next.

Objective

To demonstrate that an AI tool can be constrained to communicate the way a careful analyst would: honest about uncertainty, specific about what would confirm a theory, and free of statistical jargon when speaking to a business audience.

Process

Followed the Design Thinking cycle end to end. Empathy and Define phases produced a 10-point plan covering the bot's objective, scope, and hard boundaries. Ideate and Prototype phases turned that plan into a system prompt built in Chatbase. The Test phase went beyond the five planned scenarios: after all five passed cleanly, I added a sixth adversarial test that directly pressured the bot to abandon its core rule, and separately caught a real implementation gap where a planned disclaimer silently failed to appear, fixing it by moving the instruction into the system prompt itself.

Test results 6/6 Test scenarios passed
System rules
No confirmed causes
No statistical jargon
No fabricated claims
Try it live
Tools and technologies used

Chatbase.co (GPT-5.4 Mini), prompt engineering, adversarial test design.

Value proposition

This is the exact translation work I already do daily in retail analytics, demonstrated as a reusable, testable tool rather than a one-off explanation.

Unique value

Most people stop testing once a bot passes its planned scenarios. I treated a clean first pass as a signal my tests weren't hard enough, then designed a harder one specifically to try to break the rule that mattered most. It held. That instinct, to distrust an easy pass, is the same instinct that catches a false anomaly signal before it reaches a stakeholder.

Relevance

Directly relevant to AI/ML leadership and change management: the hardest part of deploying an AI tool in a business setting is not the model, it is getting people to trust what it tells them. This project is a small, testable example of building that trust in.

ML vs. Deep Learning: Suitability Analysis

A technical writing artifact on choosing the right model class

Introduction

Written for the graduate AI/ML Fundamentals course, this piece treats "ML or DL?" as an engineering decision rather than a matter of which sounds more advanced. It works through two real-world systems that sit on opposite ends of that decision and explains, with evidence, why each landed where it did.

Description

A researched comparative analysis examining real-time bot detection at checkout during limited collectible releases (a classic ML fit) against autonomous task learning in a consumer humanoid robot (a deep learning fit), each grounded in cited real-world implementations rather than textbook toy examples.

Objective

To demonstrate the ability to reason about model selection from first principles: identifying whether a problem's decisive signals are already condensed, human-nameable features under a hard latency constraint (favoring classic ML), or whether the problem requires learning open-ended representations from raw, high-dimensional input (favoring deep learning) — and to state where that boundary gets more nuanced than a simple rule.

Case 1 · Classic ML Bot detection at checkout
Human-nameable features
Hard real-time latency limit
Needs raw representation learning
Case 2 · Deep Learning Humanoid robot task learning
Continuous, high-dim control
Generalizes to unseen objects
Finite feature set possible
Process

Researched current, vendor-documented bot-management practices and a 2026 humanoid robotics deployment, cross-checking claims against a primary-source patent filing where available. Built the analysis around a single decision rule, tested it against both cases, then deliberately surfaced where the rule weakens, e.g. deep sequence models are used elsewhere in the same bot-detection pipeline, so the conclusion isn't overstated as absolute.

Tools and technologies used

Technical research and source verification, comparative/structured technical writing, APA citation.

Value proposition

This is the reasoning step that has to happen before any dashboard or model gets built: deciding whether a problem even calls for deep learning, or whether a simpler, cheaper, faster model already solves it. Recommending the heavier tool by default is a common and expensive mistake.

Unique value

Rather than stopping at "ML for structured data, DL for unstructured data," the analysis names the actual mechanism, whether decisive signals are already condensed and nameable versus requiring learned representations, and then stress-tests that mechanism by pointing to where it doesn't fully hold.

Relevance

Directly relevant to applied ML decision-making: before choosing an architecture, a practitioner has to justify the choice against latency, cost, and problem structure. This artifact shows that reasoning done explicitly rather than assumed.

Communicating for Learning: Explaining Neural Networks Simply

A case study in chunking, progressive disclosure, and storytelling for technical explanation

Introduction

Written for the graduate AI/ML Fundamentals course, this piece treats "explain it simply" as a skill to be tested against a real failure case rather than assumed. It starts from a bad explanation on purpose, diagnoses why it fails, and rebuilds it using specific communication-for-learning techniques.

Description

A worked comparison of two AI-generated explanations of neural networks: an unconstrained, jargon-heavy first attempt, and a deliberately staged three-part explanation built around chunking, progressive disclosure, and a single sustained analogy (recognizing a friend's knock at the door).

Objective

To demonstrate the ability to apply specific communication-for-learning strategies to a technical AI/ML concept, and to show the reasoning behind each strategy rather than just the finished explanation.

Process

Prompted Claude with no constraints first to produce a baseline, technically correct but pedagogically unusable explanation of neural networks. Diagnosed exactly which lesson principles it violated, then rewrote the prompt to explicitly require chunking, progressive disclosure, and one throughline analogy. Compared the two outputs directly and named which technique did which work.

Tools and technologies used

Claude (Anthropic), iterative prompt engineering, applied instructional-design principles (chunking, progressive disclosure, analogy-based storytelling).

Value proposition

Explaining ML concepts well to a non-technical audience is a recurring leadership task, not a one-off writing exercise. This artifact shows a repeatable method for doing it, not just one good explanation.

Unique value

Rather than presenting only the polished final explanation, this piece keeps the failed first attempt visible and names exactly why it failed. That diagnostic step, not just the fix, is what makes the technique reusable on a different concept next time.

Relevance

Directly relevant to AI/ML leadership and change management: translating technical concepts for stakeholders who don't share the vocabulary is a prerequisite for getting buy-in on any AI initiative.

Navigating Human Bias: A Retail Analytics Leadership Value Statement

A personal value statement and field-specific strategy set for leading through data bias

Introduction

Written for the graduate AI/ML Fundamentals course, this discussion post responds to a prompt on navigating human bias in AI systems as a leader. Rather than write in generic tech-leader language, I grounded it in the actual bias risks of retail data analytics: anomaly detection, customer complaint triage, and LLM-based customer-facing tools.

Description

A personal value statement on fairness in retail analytics, followed by four field-specific strategy categories addressing how human bias enters machine learning systems through data selection, labeling, interpretation of results, and self-accountability in model design.

Objective

To articulate a concrete, field-specific stance on navigating human bias in AI/ML change leadership, rather than restating the course examples in general terms, and to show how that stance translates into practical steps within an existing analytics pipeline.

Process

Reviewed three course-provided example value statements (education, healthcare, and tech leadership) for structure and tone, then rebuilt the content from my own professional context in retail data analytics rather than adapting their language. Iterated with Claude to sharpen the retail-specific framing and check the draft against the course's five named sources of bias in AI systems: data selection, labeling, algorithm design, testing and evaluation, and interpretation of results.

Value proposition

Bias in analytics tools rarely announces itself as a dramatic failure; it shows up in which anomalies get investigated and whose feedback shapes what counts as "normal." This artifact makes that risk concrete and gives it a leadership response rather than leaving it as an abstract ethics statement.

Unique value

The strategies are built around an actual analytics pipeline, not a generic checklist: historical baseline audits before trusting a model's "normal," rotating labelers to catch individual bias, and giving frontline teams a documented override channel. The testing-and-evaluation gap left visible in the toggle above is intentional; a self-audit is only credible if it names what it didn't cover yet.

Relevance

Directly relevant to AI/ML change leadership: recognizing that bias enters a system long before the algorithm runs, through data collection and labeling choices, is what separates a leader who can actually govern an AI system from one who can only audit its output after the fact.
